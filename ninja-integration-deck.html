03 ‚Äî How It Works

Four sync phases per cycle

Every 5 minutes, the sync engine runs four sequential phases. Each phase handles one data type in one direction.

1

Rev.io ‚Üí NinjaOne: Accounts

Pulls all customers from Rev.io. Checks ID mapping store ‚Äî new customers get created as NinjaOne organizations; existing ones are compared for changes (name, description). The new NinjaOne org ID is saved back to the mapping store to prevent duplicates on future runs.

Rev.io ‚Üí Ninja
2

NinjaOne ‚Üí Rev.io: Accounts

Pulls all organizations from NinjaOne. Any org without a Rev.io mapping gets created as a Rev.io customer. Named orgs only ‚Äî unnamed system orgs are skipped. Placeholder address fields satisfy Rev.io's required fields (address, city, state, zip).

Ninja ‚Üí Rev.io
3

Rev.io ‚Üí NinjaOne: Contacts

Pulls all contacts from Rev.io. Deduplicates by email address. Creates contacts in NinjaOne linked to the correct organization using the NinjaOne org ID from Phase 1. Fields mapped: first name, last name, email, phone.

Rev.io ‚Üí Ninja
4

NinjaOne ‚Üí Rev.io: Contacts

Pulls all contacts from NinjaOne. Creates new contacts in Rev.io linked to the corresponding customer. Email used as deduplication key across both systems.

Ninja ‚Üí Rev.io
04 ‚Äî Project Structure

What's in the codebase

üîå

src/revio.js

Rev.io API client. Handles API key exchange for a JWT bearer token (auto-refreshes), full cursor-based pagination for customer and contact endpoints, and all CRUD operations.

üõ°Ô∏è

src/ninjarmm.js

NinjaOne OAuth 2.0 client. Client credentials flow ‚Äî exchanges client ID + secret for a bearer token. Handles org and contact create/read endpoints on the NinjaOne REST API.

üó∫Ô∏è

src/mapping.js

Field mapping logic in both directions. Translates Rev.io customer fields to NinjaOne org fields (and back), and Rev.io contact fields to NinjaOne contact fields. Diff detection for update decisions.

üíæ

src/state.js

ID mapping store persisted to data/sync-state.json. Tracks which Rev.io customer ID maps to which NinjaOne org ID (and vice versa) so records are never duplicated across sync runs.

‚öôÔ∏è

src/sync.js

The core sync engine. Orchestrates all four phases, collects stats (created / updated / skipped / errors), and logs results. Can be run as a one-shot script or called by the server scheduler.

üñ•Ô∏è

src/server.js + public/

Express web server with a live dashboard. Uses Server-Sent Events (SSE) to stream sync logs to the browser in real time. Shows current status, last sync time, record counts, and a sync-now button.

08 ‚Äî Next Steps

To reach production

1

Fix contact payload (1-line change)

Remove tags field from NinjaOne contact create payload ‚Äî NinjaOne only accepts {firstName, lastName, email, organizationId}. Quick fix, then re-run sync to confirm contacts flow through cleanly.

2

Demo walkthrough

Create a customer in Rev.io ‚Üí verify it appears in NinjaOne within 5 minutes. Create a contact in NinjaOne ‚Üí verify it appears in Rev.io. Run npm start, open the live dashboard, trigger a sync, walk through the results.

3

Production considerations

Replace polling (5-min interval) with webhooks for real-time sync. Add proper conflict resolution (timestamp-based, not last-write-wins). Deploy as a persistent service (systemd, Docker, or cloud function). Add retry logic and alerting for failed records.

4

Extend to tickets / work orders

Phase 2 of the integration: sync NinjaOne tickets/alerts into Rev.io work orders. This is the deeper value ‚Äî service desk workflow unification. Requires additional NinjaOne scopes and Rev.io ticketing API access.
