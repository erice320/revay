⚠️ Data Quality Notice — Read Before Drawing Conclusions

This analysis is based on Wrike time log data (Aug 2025–Jan 2026). The categorization system works by mapping each timelog to a task, then tracing the task's parent folder to a client project. Three known structural gaps inflate Admin Time and suppress utilization metrics:

1. Orphaned Tasks → Admin Time
Any task logged outside a Wrike project folder (e.g. "DE Client Calls") silently falls into Admin Time. Client-facing work becomes invisible in utilization. Estimate: 275h+ affected.
2. "Other" (23%) = Likely Structure Issues
The 1,822h "Other" bucket likely represents tasks in deep subtask hierarchies where the project folder walk failed — not truly unclassifiable work. Some of these hours may be misattributed to the wrong project entirely.
3. Capacity Model Has Uncertainty Range
Per-client hour averages (9h PSA Web, 43h Legacy) are derived from the same data with the above gaps. Hire counts should be treated as estimates with ±15–20% uncertainty until data quality is improved.
Prerequisite action: Before making headcount or performance decisions from this data, audit Admin Time at the task level to identify productive work currently hidden there. See Opportunity #0 in the improvement table below.

Key Findings for Leadership Discussion

🔴
Project Control consumes 34% of all client-facing hours 2,671 of 7,862 project hours are logged as "Project Control" — coordination, status updates, project management overhead. This is the single largest task category, exceeding Data Load, Training, and Kickoffs combined. If even half of this is waste, recovering it would increase effective delivery capacity by 17%.
🔴
Two full-time ICs log 21% and 26% utilization as measured Alexander Hills and Derrick James are full-time ICs on the legacy Billing team. Even adjusting for "DE Client Calls" being miscategorized as Admin Time, true utilization is ~37–39%. Both log significant Internal Meetings (171h, 230h). This is recoverable capacity — potentially 300–400 billable project hours per 6-month period per person.
🟡
The PSA Web team is running near capacity with just 2 active SAs Nicole and Kegan average 165 combined project hours/month. At ~3.5h/month per active client, this supports approximately 47 concurrent clients. They currently have 95 active/on-hold/off-track accounts — indicating they are over their true delivery capacity, which explains the engagement drop-off on on-hold accounts.
🟡
Lunch is logged as Admin Time across the team — inflating overhead metrics Alexander Hills and Derrick James alone logged 220h as "Lunch." Across the full team this likely adds 400–500h to "Admin Time" totals over 6 months, artificially depressing utilization rates. This is a time tracking hygiene issue that makes the data misleading for management.
🟡
"DE Client Calls" (275h) lands in Admin Time due to a Wrike folder structure issue — not a labeling error Client-facing calls are logged on tasks that live outside any project folder in Wrike, so the categorization engine routes them to Admin Time. The fix is structural: move "DE Client Calls" tasks into client project folders in Wrike so they're picked up automatically. A surface-level reclassification in reporting will recur every period without fixing the root cause. For Alexander Hills (149h) and Derrick James (127h) this meaningfully understates their apparent performance — and there may be additional productive hours in Admin Time beyond this category if other tasks are also orphaned from project folders.
🟢
JD Caulkins, Marshall Harris, and Stephanie Okada are the delivery engine These three combined log 2,396 project hours (76–84% utilization) across the 6-month period — 30% of all project work from 17% of headcount. They are the team's highest-leverage contributors on the legacy Billing side.
🟢
PSA Web can double its graduation rate before needing another SA hire — IF product unblocks clients 20 clients are on hold for functionality. If Apollo 1 deliveries unlock even half, Nicole and Kegan can absorb the re-activation without new headcount in the short term. The constraint is product, not people.
